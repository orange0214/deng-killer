# 小中老三登 Killer PRD v0.1

## 1. 背景与目标

用户在日常交流、饭局、会议或辩论中，经常会听到他人使用听起来很权威但可能并不准确的事实性陈述。由于对方可能具有年龄、学历、职位、权力或表达气势上的优势，用户容易在当下轻信，事后才发现对方使用了错误案例、错误因果或过度简化的事实。

本产品的目标不是帮助用户“赢辩论”，而是作为一个低打扰的事实核验助手，帮助用户在对话过程中或对话结束后识别值得核验的事实性主张，发现可能错误的信息，并给出可靠依据。

## 2. 产品定位

第一版产品定位为：

> 对话中的事实护盾：实时记录对话，抽取可核验事实主张，对可疑陈述进行证据核验，并向用户提供低打扰提醒、错误原因和证据摘要。

核心价值：

- 帮助用户抵抗权威感、语气自信和信息不对称带来的误信。
- 让用户知道哪些话值得查，而不是盲目接受或盲目反驳。
- 将事实核验结果转化为可用于真实对话的温和回应。

## 3. 第一版范围

### 3.1 包含功能

第一版 MVP 包含以下能力：

1. 录音并将语音转写为文字。
2. 从转写文本中抽取事实性、陈述性、可核验的主张。
3. 判断每条主张是否值得进入核验流程。
4. 对值得核验的主张进行初步分类。
5. 对可疑或重要主张调用搜索工具和模型进行证据核验。
6. 输出核验结论、错误原因、证据摘要和更准确说法。
7. 以低打扰方式提醒用户发现了可能错误的事实陈述。

### 3.2 暂不包含功能

第一版暂不追求：

- 完全实时打断式纠错。
- 对所有对话内容做全量事实判断。
- 直接给出攻击性辩论话术。
- 微调一个只输出 yes/no 的真伪判断模型。
- 对政治、医疗、法律、投资等高风险领域给出最终决策建议。

## 4. 用户流程

### 4.1 对话记录

用户开启一次对话记录。系统采集语音，并通过 ASR 模型将语音转写为带时间戳的文本。

基础输出：

```json
{
  "speaker": "A",
  "start_time": "00:01:12",
  "end_time": "00:01:18",
  "text": "FastAPI 比 Django 快是因为 FastAPI 是多线程。"
}
```

### 4.2 事实主张抽取

系统不处理所有句子，而是从转写文本中抽取可核验主张。

需要抽取的内容示例：

- “FastAPI 比 Django 快是因为 FastAPI 是多线程。”
- “Redis 是关系型数据库。”
- “某公司去年利润增长了 30%。”
- “美国总统任期是 5 年。”

不优先处理的内容示例：

- “我觉得 Python 写起来更舒服。”
- “这个人不太靠谱。”
- “这个东西很高级。”
- “以后 AI 肯定会取代所有程序员。”

### 4.3 是否值得核验

系统对每条候选主张进行 claim worthiness 判断，判断其是否值得进入核验流程。

判断维度：

- 事实性：是否在表达事实，而不是观点或情绪。
- 可验证性：是否可以通过公开资料或权威来源验证。
- 具体性：是否包含明确主体、谓词、时间、数量、因果或定义。
- 重要性：该主张是否可能影响用户判断。
- 风险性：该主张是否可能误导用户或被权威感包装。

输出格式：

```json
{
  "claim": "FastAPI 比 Django 快是因为 FastAPI 是多线程。",
  "is_checkworthy": true,
  "claim_type": "technical_fact",
  "verifiability": "high",
  "importance": "medium",
  "checkworthy_score": 0.86,
  "reason": "该陈述包含明确的技术因果关系，可以通过框架文档和技术资料核验。"
}
```

推荐阈值：

- 0.00 - 0.40：不核验。
- 0.40 - 0.70：记录，暂不打扰。
- 0.70 - 1.00：进入核验队列。

### 4.4 初步分类

系统不使用简单 yes/no 判断，而是对主张进行初步分类。

分类标签：

- `likely_true`：初步看基本正确。
- `possibly_false`：初步看可能错误。
- `uncertain`：证据不足或模型不确定。
- `needs_context`：需要更多上下文。
- `controversial`：存在争议，不能简单判定。

输出格式：

```json
{
  "claim": "FastAPI 比 Django 快是因为 FastAPI 是多线程。",
  "initial_label": "possibly_false",
  "confidence": 0.72,
  "reason": "该陈述将 FastAPI 的性能优势归因于多线程，可能存在技术因果错误。"
}
```

初步分类只代表可疑程度，不作为最终真伪结论。

### 4.5 证据核验

当主张满足以下条件之一时，进入证据核验：

- 初步分类为 `possibly_false`。
- 初步分类为 `uncertain` 且重要性较高。
- 初步分类为 `needs_context` 且可通过补充资料澄清。
- `checkworthy_score` 较高但模型置信度不足。

核验流程：

1. 将复杂主张拆解成更小的原子主张。
2. 为每个原子主张生成搜索查询。
3. 调用搜索工具或权威文档源。
4. 抽取证据片段。
5. 判断证据与主张的关系。
6. 汇总最终结论。

复杂主张拆解示例：

原句：

```text
FastAPI 比 Django 快是因为 FastAPI 是多线程。
```

拆解：

```text
A: FastAPI 在某些场景下可能比 Django 快。
B: FastAPI 快的原因是它是多线程。
```

A 可能部分成立，B 可能不准确。

### 4.6 最终输出

最终输出应包含：

- 原始陈述。
- 核验结果。
- 置信度。
- 问题点。
- 更准确说法。
- 证据摘要。

示例：

```json
{
  "original_claim": "FastAPI 比 Django 快是因为 FastAPI 是多线程。",
  "verdict": "possibly_false",
  "confidence": 0.82,
  "issue": "该陈述将 FastAPI 的性能优势归因于多线程，原因不准确。",
  "correction": "FastAPI 在某些高并发 I/O 场景下表现较好，主要原因通常与 ASGI、Starlette 和 async/await 异步处理有关，而不是框架本身是多线程。",
  "evidence_summary": [
    "FastAPI 基于 Starlette，并支持 ASGI。",
    "FastAPI 支持 async/await 异步请求处理。",
    "多线程更多与部署服务器或运行环境有关，不是 FastAPI 的核心设计原因。"
  ]
}
```

面向用户的展示文案：

```text
发现 1 条可能错误的事实陈述。

原话：
FastAPI 比 Django 快是因为 FastAPI 是多线程。

核验结果：
可能错误。

原因：
FastAPI 的性能优势通常主要来自 ASGI 和异步 I/O，而不是框架本身是多线程。
```

## 5. 系统方案

### 5.1 模块划分

第一版系统由以下模块组成：

1. Audio Capture：录音与音频分段。
2. ASR：语音转文字。
3. Claim Extraction：事实主张抽取。
4. Claim Worthiness Scoring：是否值得核验评分。
5. Initial Classification：初步可疑程度分类。
6. Claim Decomposition：复杂主张拆解。
7. Search and Retrieval：搜索与资料检索。
8. Evidence Judgement：证据立场判断。
9. Verdict Generation：最终结论生成。
10. User Notification：用户提醒与展示。

### 5.2 推荐第一版技术路线

第一版建议采用 iOS 优先的混合式架构：

- 客户端：Swift 原生 iOS App。
- 后端：Python + FastAPI 核验服务。
- 检索：搜索优先，轻量 RAG 辅助。
- 实时策略：完整句子触发，而不是固定 30 秒批处理。
- 隐私策略：完整音频、完整转写和会话复盘只保存在用户手机本地；后端只接收需要核验的单条事实主张。

第一版不建议直接微调模型，而是优先使用通用模型、搜索工具和轻量 RAG 验证产品价值。

推荐实现：

- iOS Audio Capture：使用 `AVAudioEngine` 进行录音、分段和音量检测。
- ASR：优先使用 Apple Speech 或端侧 Whisper 类模型生成转写。
- Sentence Finalization：ASR partial transcript 不触发核验，只有完整句子才进入后续流程。
- Local Storage：手机本地保存音频、完整转写、会话复盘和核验结果。
- Claim Extraction：优先在本地或后端使用通用 LLM 结构化输出 JSON。
- Worthiness Scoring：LLM 分类 + 规则过滤 + 阈值策略。
- Initial Classification：通用 LLM 或轻量分类模型。
- Evidence Verification：搜索工具 + 轻量 RAG + LLM 证据归纳。
- Backend Storage：默认不保存完整对话、完整转写、音频、speaker 历史或普通 claim；最多保存用户授权后的高置信错误样例和公开证据缓存。

后续当积累足够数据后，再考虑微调小模型：

- 优先微调“事实主张抽取”和“是否值得核验”模型。
- 不优先微调只输出 yes/no 的最终真伪模型。
- 最终真伪判断必须保留证据链，而不是完全依赖模型直觉。

### 5.3 手机端与后端职责

第一版应明确区分本地数据和后端数据，降低隐私风险。

iOS 端职责：

- 录音、权限管理、ASR 转写和完整句子结束检测。
- 本地保存完整会话、音频、转写、核验结果和会后复盘。
- 对低价值句子进行本地过滤，不上传无关闲聊。
- 只将高价值事实主张上传后端核验。
- 展示低打扰实时提醒和会后复盘。
- 支持用户删除单次会话或设置自动删除周期。

后端职责：

- 接收单条或少量事实主张，而不是完整 transcript。
- 执行 claim worthiness、初步分类、主张拆解、搜索/RAG 检索、证据判断和结论生成。
- 默认请求处理完成后丢弃 claim 正文和上下文。
- 服务日志禁止记录请求正文、claim 文本、完整上下文和模型输入。
- 仅保存非内容型运行元数据，例如 request id、耗时、状态码、模型版本。
- 只有在用户授权时，才保存高置信错误 claim 的最小化记录。

### 5.4 完整句子触发核验

第一版不使用固定 30 秒批处理。系统在 ASR 判断一句话结束后，立即进入轻量处理流程。

处理流程：

1. iOS 端持续录音，ASR streaming 输出 partial transcript。
2. 当 ASR 判断一句话结束，生成 finalized sentence。
3. iOS 端对该句进行基础过滤，明显闲聊、情绪和观点句不上传。
4. 高价值事实 claim 上传后端。
5. 后端先进行 claim extraction 和 checkworthy scoring。
6. 低分主张不核验；中分主张进入会后复盘；高分主张进入即时核验队列。
7. 初步分类为 `possibly_false`、`uncertain`、`needs_context` 或重要性高的主张才触发搜索/RAG。
8. App 可以先展示“正在核验”，最终结果返回后再进行提醒。

推荐延迟目标：

- 句子结束后 1-2 秒内完成事实主张抽取和值得核验判断。
- 3-8 秒内完成缓存或轻量 RAG 命中核验。
- 8-20 秒内完成搜索增强核验。
- 超过 20 秒的任务进入会后复盘，不强行实时提醒。

### 5.5 搜索与 RAG 方案

第一版采用 Hybrid Retrieval：搜索优先，轻量 RAG 辅助。

不建议只使用 RAG：

- 用户对话中的事实范围不可预测，自建 RAG 无法覆盖开放世界。
- 新闻、职位、价格、版本、政策、公司数据等内容会变化，必须依赖搜索。
- 过期 RAG 内容可能产生“看起来有依据但其实旧了”的误导。

不建议只使用搜索：

- 搜索结果噪声高，延迟和质量不稳定。
- 技术文档、标准规范、定义类知识更适合用 RAG 加速和控源。
- 已核验过的公开证据可以缓存，避免重复搜索。

RAG 数据库只存三类内容：

- Curated Sources：官方文档、标准文档、可信百科、常见技术框架文档。
- Evidence Cache：历史核验中用过的公开网页、摘要、来源、抓取时间和证据立场。
- Verification Metadata：不含完整对话的核验元数据，例如领域、标签、模型版本、证据来源和结果质量反馈。

RAG 数据库不存：

- 用户完整对话。
- 用户完整转写。
- 原始音频。
- speaker 身份或对话历史。
- 未经用户授权的普通 claim。
- 用户 claim 的长期 embedding。

推荐存储：

- PostgreSQL：公开来源、证据元数据、授权后的错误样例、运行元数据。
- pgvector：公开资料 chunk 和公开证据 chunk 的向量检索。
- 第一版暂不需要 Milvus、Qdrant 等独立向量数据库。

检索策略：

- 稳定技术事实、定义、官方文档类 claim：优先 RAG，再补搜索。
- 时间敏感 claim：优先搜索，不信任旧 RAG。
- 高风险 claim：搜索 + RAG 都查，证据不足时输出不确定。
- 相似历史错误：先查授权后的错误样例和公开证据缓存，再决定是否重新搜索。

### 5.6 后端 API 设计

第一版 API 围绕单条 claim，而不是完整会话。

`POST /verify-claim`

- 输入：单条 claim、可选最小上下文、语言、领域、实时性标记。
- 输出：verdict、confidence、correction、evidence_summary、sources。
- 默认无持久化。

`POST /verify-claim/stream`

- 返回核验进度：`received`、`classifying`、`retrieving`、`judging_evidence`、`completed`。
- 不保存事件正文。

`POST /feedback`

- 输入：result_id、本地生成的匿名会话 id、用户反馈。
- 默认不上传完整对话。
- 用户可以选择是否附带 claim 文本帮助改进。

`POST /opt-in/error-examples`

- 仅当用户主动同意时，上传高置信错误样例用于产品改进。
- 上传内容必须最小化，不包含完整上下文、音频或 speaker 身份。

### 5.7 正确性保障策略

第三步和第四步无法保证绝对正确，只能通过流程设计降低错误率。

策略：

- 使用结构化输出，避免模型自由发挥。
- 使用多标签分类，而不是 yes/no。
- 对每个判断输出置信度和原因。
- 对复杂主张进行拆解，避免一句话整体误判。
- 对可疑主张强制进入搜索和证据核验。
- 将最终结论建立在证据链上。
- 对低置信度结论展示为“证据不足”或“需要上下文”。
- 在不上传完整对话的前提下记录用户反馈，用于后续评估和模型优化。

## 6. 关键体验原则

### 6.1 低打扰

系统不应频繁打断真实对话。默认只做轻量提醒，例如震动、角标或会后总结。

实时提醒只用于：

- 高置信度错误。
- 高重要性事实。
- 用户主动开启实时辩论辅助模式。

### 6.2 可解释

用户需要知道系统为什么这样判断。每条核验结果都应包含：

- 原始陈述。
- 判断标签。
- 证据摘要。
- 不确定性说明。
- 来源或依据。

## 7. 成功指标

第一版可以用以下指标评估：

- ASR 转写可读率。
- 可核验主张抽取准确率。
- 值得核验判断准确率。
- 可疑事实召回率。
- 错误提醒误报率。
- 用户是否认为结果有帮助。
- 用户是否愿意在真实对话后查看复盘。

## 8. 风险与约束

### 8.1 隐私风险

产品会记录真实对话，必须明确告知用户录音与转写行为。第一版默认完整音频、完整转写和会后复盘只保存在用户手机本地，后端不保存完整对话内容。

隐私约束：

- 后端不保存完整音频、完整转写、speaker 对话历史或普通 claim。
- 后端默认不记录请求正文、claim 文本、完整上下文或模型输入。
- 后端最多保存用户授权后的高置信错误 claim、纠正说明、证据来源和证据摘要。
- RAG 向量库只保存公开资料和公开证据缓存，不保存用户对话。
- 用户应能删除本地会话，并配置本地数据自动删除周期。
- 后续如果要使用用户数据微调模型，必须单独设计明确授权、脱敏、导出和删除机制。

### 8.2 社交风险

如果提醒过于攻击性，产品可能鼓励用户在对话中对抗他人。第一版应强调事实澄清和自我保护，而不是辩论胜负。

### 8.3 模型幻觉风险

模型可能错误分类、错误归纳证据或生成不存在的依据。因此最终输出必须绑定搜索结果或权威来源，并对不确定内容明确标注。

### 8.4 领域风险

医疗、法律、金融、政治等领域的事实核验更复杂，第一版应避免输出确定性决策建议。

## 9. 第一版验收标准

第一版完成后，应能跑通以下场景：

1. iOS 端录音并在本地生成完整句子级转写。
2. 系统只将高价值事实 claim 上传后端，而不是上传完整对话。
3. 后端识别该 claim 是否值得核验。
4. 后端对可疑 claim 进行初步分类。
5. 后端对至少一条 claim 进行搜索或 RAG 资料核验。
6. 系统输出核验结果、错误原因、证据摘要和更准确说法。
7. 对低置信度结果，系统能够输出“证据不足”或“需要上下文”，而不是强行判断。
8. 后端数据库和日志中不出现完整对话、完整转写、音频或 speaker 对话历史。
9. 普通 claim、正确 claim 和低置信 claim 默认不被后端持久化。
10. 用户删除本地会话后，手机端音频、转写、核验结果和复盘全部删除。

## 10. 示例端到端流程

输入对话：

```text
A: FastAPI 比 Django 快是因为 FastAPI 是多线程。
B: 是吗？我之前以为是异步。
```

系统抽取：

